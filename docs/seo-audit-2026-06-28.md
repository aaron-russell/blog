# SEO audit: aaron-russell.co.uk

Audit date: 28 June 2026  
Scope: production site, repository, Cloudflare delivery, content, accessibility, structured data, competitors, and AI-search readiness.

Remediation update: this worktree now includes the merged `master` canonical fix plus follow-up code changes for sitemap `lastmod`, tag normalization, RSS GUID normalization, and social metadata cleanup. Production findings below still describe the audited live site until those changes are deployed.

## Executive summary

The site has a sound technical base: all 28 sitemap pages returned `200`, HTTPS and `www` redirect in one hop, every indexed page has a single H1, canonical, description, Open Graph/Twitter metadata, and parseable JSON-LD, and no sitemap page is orphaned or deeper than two clicks. All 72 fallback images and 26 unique social images loaded successfully. Lighthouse SEO and accessibility both scored 100 on the homepage and newest article.

The limiting factor is not crawlability. It is the mismatch between the site's strong modern presentation and its small, stale, diffuse body of content. Twenty-four of 25 posts predate 2024, while the homepage claims an AI/web/Cloudflare focus that the archive does not yet substantiate. There is no dedicated About/profile page, no visible GitHub/LinkedIn proof, no topic hub architecture, and most articles are generic rather than experience-led. This suppresses topical authority, linkability, and AI citation potential.

The immediate technical priorities are also concrete:

1. Fifteen article canonicals omit the enforced trailing slash and therefore point to a `308` URL rather than the `200` sitemap URL. This is fixed in the current branch after merging `master`.
2. The PostHog bundle ships on every page even when tracking cannot run; Lighthouse found 58.5 KB of its 68.5 KB payload unused. Fixed in code on this branch by loading PostHog only after tracking checks pass.
3. Cloudflare JavaScript Detection consumed about 5.1 seconds of scripting in the headless article run. Headless automation is deliberately challenged, so this is not proof of equivalent real-user cost, but it warrants a field-data review.
4. The article hero is visibly distorted on mobile: Lighthouse measured a displayed ratio of `378 × 508` against an intrinsic ratio of `768 × 512`. Fixed in code on this branch with responsive aspect-ratio and `object-fit`.
5. Thirteen titles exceed 60 characters and 15 descriptions exceed 160; one Agile description repeats its entire paragraph.

The highest-return strategy is to fix those template defects, publish a credible author/profile surface, and then concentrate on three evidence-backed clusters: Angular/Firebase, Astro/Cloudflare, and AI-assisted product engineering. The archive should support those clusters, not attempt to be equally authoritative across every developer topic.

### Verified defects

#### Canonicals point through a redirect

Status: fixed in code on this branch via the merged `master` change. Deploy still required.

Fifteen article pages use a Contentful canonical without the trailing slash, while Astro enforces `trailingSlash: 'always'`. Example:

- Page and sitemap: `/blog/angular-vs-react-vs-vue-choosing-the-right-javascript-framework-for-you/`
- Canonical: `/blog/angular-vs-react-vs-vue-choosing-the-right-javascript-framework-for-you`
- Canonical response: `308` to the slash URL

Affected posts are the newest 15, from Pit Crew through 10 Best VS Code Extensions. The ten older posts self-canonicalize correctly. This is not a catastrophic loop, but it sends conflicting canonical/sitemap/internal-link signals and wastes a hop. Normalize the CMS value before using it in canonical, `og:url`, JSON-LD, and RSS GUID.

#### Sitemap has no freshness data

Status: fixed in code on this branch. Posts now emit `lastmod`, and home/blog inherit the latest substantive post update timestamp.

Every sitemap entry contains only `<loc>`. Add accurate `<lastmod>` using the significant content modification date. Google states that it uses `lastmod` when it is consistently accurate; do not update it for deploy-only or copyright changes.

#### Topic taxonomy is dirty and has no hubs

The archive exposes query filters but no indexable topic pages. Tag variants include:

- `Web Development` / `Web development`
- `My Story` / `My story`
- `Pagespeed` / `pagespeed`
- typo: `Quesrions`

Status: fixed in code on this branch for the site layer. Known variants are normalized at content ingestion, and curated topic hubs now exist at `/topics/angular/`, `/topics/cloudflare/`, and `/topics/performance/`. CMS cleanup is still recommended so Contentful remains the source of truth.

Normalize tags in Contentful as the source of truth. Then create a small number of curated static hub pages such as `/topics/angular/`, `/topics/cloudflare/`, and `/topics/performance/`, each with an introduction and ordered reading path. Do not generate thin pages for every tag.

#### External link needing review

`https://zebrahardware.co.uk/`, linked from `/blog/improve-your-seo/`, returned `500`. Review or replace it. Eighteen `403` results were anti-bot responses and need manual browser checks before removal.

### CSP and analytics conflict

Lighthouse logged CSP errors because Cloudflare injects `static.cloudflareinsights.com/beacon.min.js` but the policy does not allow it. Choose one analytics system:

- If Cloudflare Web Analytics is needed, add the exact Cloudflare script/connect sources and test with Report-Only first.
- If PostHog or edge analytics is sufficient, disable the automatic Cloudflare beacon rather than weakening CSP.

Do not add Zaraz merely because it exists. With one small analytics integration, it adds operational complexity without a measured SEO benefit.

### Cloudflare caching recommendations

- Do not buy or enable APO for this Astro/Pages site. It targets WordPress and is not the missing layer here.
- HTML edge caching is optional, not urgent: TTFB was already 78–92 ms. If enabled, cache only anonymous GET/HEAD HTML and bypass `/contact/`, form submissions, `/cdn-cgi/`, preview routes, and any future personalized content.
- Static JS/CSS/fonts already receive long-lived caching and Brotli; leave this alone.
- Do not add Polish or Cloudflare Image Resizing for Contentful images. Contentful already emits resized AVIF/WebP/JPEG variants; double transformation is redundant.
- Mirage is deprecated. The site already uses native lazy loading, `picture`, and `srcset`.
- Early Hints may help the two small render-blocking CSS files, but Lighthouse's larger cost comes from scripting. Test after the JS work; do not prioritize it first.
- Smart Tiered Cache/Cache Reserve will not materially improve a 28-page static blog with sub-100 ms TTFB.

### Primary performance causes

1. **Cloudflare JavaScript Detection:** `cdn-cgi/challenge-platform/scripts/jsd/main.js` accounted for 5.1 seconds of scripting in the article run. Cloudflare explicitly warns that automation/headless browsers can trigger bot detection, so the magnitude is probably lab-inflated. Review Cloudflare Web Analytics field data before changing security. If JS Detection is not enforced by a WAF rule, its site-wide injection has questionable value. [Cloudflare JSD documentation](https://developers.cloudflare.com/cloudflare-challenges/challenge-types/javascript-detections/)
2. **PostHog shipped eagerly:** the 68.5 KB analytics chunk was 85% unused, with an estimated 58.5 KB saving. The static `import posthog from 'posthog-js'` downloads it before `shouldTrack()` can reject tracking. Fixed in code on this branch with a dynamic import.
3. **Render-blocking CSS:** Lighthouse estimated 1.55 s on home and 420 ms on the article under simulated mobile conditions. The files are only about 7–8 KB transferred, so first remove JS pressure, then test CSS inlining/splitting.
4. **Article hero distortion:** CSS forces the hero to `height: min(62vh, 720px)` without `object-fit`. Lighthouse reports the 1.50 intrinsic image rendered at 0.74 on mobile. This is a UX defect and can make the LCP image needlessly tall. Fixed in code on this branch.
5. **No field RUM available to this audit:** enable one coherent RUM source and use p75 LCP/INP/CLS segmented by template before making high-cost tuning decisions.

### Astro-specific recommendations

- Keep static output and partial hydration. There are no framework islands to remove; the architecture is already mostly HTML/CSS.
- Dynamically import PostHog only after consent/key/DNT checks.
- Keep the existing responsive Contentful image component; it already emits AVIF, WebP, JPEG, dimensions, `srcset`, `sizes`, lazy loading, and high-priority hero images.
- Fix hero display with `object-fit: cover` and an explicit responsive aspect ratio.
- Consider Astro content loaders/content collections only if they add schema validation around Contentful data. A CMS rewrite offers no SEO return.
- View Transitions and Server Islands do not solve a measured SEO problem here. Do not add them for this audit.
- Prerendering is already correct. Edge rendering would make caching and reliability worse for content that changes only at publish time.

## Implementation examples

### Canonical normalization

```ts
export const canonicalUrl = (value: string) => {
  const url = new URL(value)
  if (!url.pathname.endsWith('/')) url.pathname += '/'
  url.search = ''
  url.hash = ''
  return url.toString()
}

const canonical = canonicalUrl(
  post.canonical || `${author.website}/blog/${post.slug}/`
)
```

Use the same normalized value in `<link rel="canonical">`, `og:url`, `BlogPosting.url`, `mainEntityOfPage`, RSS GUID, and sitemap.

### Load PostHog only when tracking is allowed

```ts
if (shouldTrack()) {
  const { default: posthog } = await import('posthog-js')
  posthog.init(posthogApiKey, {
    api_host: 'https://eu.i.posthog.com',
    autocapture: false,
    capture_pageleave: false,
    capture_pageview: false,
  })
  posthog.capture('$pageview', { path: window.location.pathname })
}
```

### Preserve article hero proportions

```css
.image {
  aspect-ratio: 3 / 2;
  height: auto;
  object-fit: cover;
}

@media (min-width: 1024px) {
  .image {
    max-height: 720px;
  }
}
```

### Sitemap freshness

```ts
const entries = posts.map((post) => ({
  loc: toAbsoluteUrl(`/blog/${post.slug}/`),
  lastmod: post.rawUpdatedDate || post.rawDate,
}))
```

Only expose `lastmod` when those timestamps represent significant editorial changes.
